# 02 — Source-of-truth pattern

## Why Python (not TS / JSON)

Один Python-файл драйвит ОБА слоя — TS mock и Supabase blocks. Это гарантирует, что
mock и DB не разъедутся (типичный баг при ручном дублировании).

## Файл `/tmp/lessonN_steps.py`

Шаблон в `templates/lesson_steps.py.tmpl`. Структура:

```python
TITLE = "Подготовка Mac mini к роли AI-сервера"
DURATION = "45 мин"
DESCRIPTION = "Длинное описание для шапки..."

OUTCOME = [
    "Что получишь — пункт 1",
    "Что получишь — пункт 2",
    ...
]

SKILLS = ["Mac mini", "AI-native setup", ...]

PROMPTS = [
    # (id, label, text, featured)
    (1, "промпт для AI-помощника",
     "Текст промпта с \\n переносами\n"
     "Можно много строк через конкатенацию.",
     True),
    (2, "следующий промпт", "...", False),
]

REPOS = [
    ("<your-github-org>/intensive-bonus-mac-mini", "описание репо"),
]

TIMECODES = [
    ("00:00", "Раздел 1"),
    ("04:00", "Раздел 2"),
    # для бонусов TIMECODES всё равно НЕ показываются
    # (см. 05-frontend-rules.md), но оставляй их для парности структуры
]

# (num, title, goal, body, command_or_None, command_title_or_None, resources_or_None)
STEPS = [
    (1,
     "Заголовок шага",
     "Цель — что получим в результате",
     "Что делаем — детали, инструкции, объяснения",
     None,  # или "bash command\nна нескольких строках"
     None,  # или "Заголовок code-блока"
     None,  # или [{"label": "...", "href": "...", "kind": "tool|repo|doc"}, ...]
    ),
    ...
]
```

## Конвенции

### TITLE / DURATION / DESCRIPTION
- **TITLE** — короткий, 5-7 слов, без emoji.
- **DURATION** — формат `"45 мин"` или `"30 мин"`.
- **DESCRIPTION** — 2-4 предложения. Объясни, зачем это нужно, и упомяни ключевые шаги.
  Можно ссылаться на «кнопку выше» (Скопировать всё в Markdown), если урок предполагает
  использование AI-помощника.

### OUTCOME (что получишь)
- 4-7 пунктов
- Каждый — конкретный результат, не процесс
- Не «настроишь X», а «X работает 24/7»

### SKILLS
- 5-10 тегов
- Короткие (1-3 слова)
- Реальные технологии или процессы, не абстракции

### PROMPTS
- (id, label, text, featured)
- `featured=True` для основного промпта (выделяется золотым)
- Только ОДИН promptik с `featured=True` на урок (визуально)
- Используй `\\n` (двойной слэш) когда внутри строки нужен литеральный `\n` для отображения

### REPOS
- Только публичные или те, к которым у юзера будет доступ
- Описание короткое (5-10 слов)

### STEPS — главное

Структура шага: **Шаг N → Заголовок → Цель → Что делаем → (опц.) код → (опц.) ресурсы**

- **num** — порядковый номер. Начиная с 1, монотонно возрастает.
- **title** — что делаем одной фразой. Не пиши «Установка X» — пиши «Установи X».
- **goal** — зачем (мотивация / результат шага). 1 предложение.
- **body** — детали. Можно несколько абзацев через `\\n\\n`. Markdown `**bold**` поддерживается.
- **command** — bash-команды если нужны. Несколько строк через `\\n`. Не используй здесь
  shell-меты, которые нужно эскейпить — пиши по-человечески, юзер копирует.
- **command_title** — короткий заголовок для code-блока (5 слов max).
- **resources** — список tool/repo/doc ссылок. Каждая = `{"label", "href", "kind"}`.
  `kind`: `tool` (золотой), `repo` (тёмный), `doc` (серый).

### Один шаг = одно действие

Если шаг содержит «и потом» — раздели на два. Шагов 10-20 норм. Меньше — слишком абстрактно.
Больше — юзер устаёт.

### Code-блок присоединён к шагу

Если у шага есть `command` — генератор автоматически добавляет `code` блок СРАЗУ ПОСЛЕ
step-блока в DB. На странице это рендерится как visually-grouped пара.

Не делай отдельных code-блоков вне шагов (только если код — это вспомогательный snippet,
а не часть инструкции).

## Шаблоны промптов для разных задач

См. `templates/lesson_steps.py.tmpl` — там есть готовый skeleton с заполнителями `<...>`.
